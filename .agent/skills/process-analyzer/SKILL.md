# 프로세스 분석기

이 스킬은 프로세스 문서(PRC)의 구조적 완전성과 품질을 분석하여 개선이 필요한 부분을 식별합니다.

## 사용 시점

이 스킬을 사용하는 경우:
- 기존 프로세스 문서의 품질 점검
- 프로세스 개선 전 현황 분석
- 문서 구조의 완전성 검증
- 프로세스 표준 준수 여부 확인

## 지침

### 1단계: 문서 식별

1. 분석 대상 문서 확인
   - docId가 `PRC-` 로 시작하는지 확인
   - 파일 위치 및 접근 가능 여부 확인

2. 메타데이터 추출
   - YAML frontmatter 파싱
   - docId, title, owner, status, version 확인

### 2단계: 구조 완전성 분석

다음 필수 섹션의 존재 여부를 확인합니다:

| 섹션 | 필수 | 확인 내용 |
|------|------|----------|
| 목적 | O | 프로세스가 해결하는 문제가 명확한가? |
| 적용 범위 | O | 대상 조직/역할/상황이 정의되었는가? |
| 입력/출력 | O | 시작 조건과 산출물이 명시되었는가? |
| 역할 (RACI) | O | 모든 활동에 책임자가 지정되었는가? |
| 프로세스 흐름 | O | 단계별 절차가 있는가? |
| 예외 처리 | O | 예외 상황과 에스컬레이션이 정의되었는가? |
| KPI | △ | 측정 지표가 있는가? (권장) |

### 3단계: RACI 매트릭스 검증

RACI 매트릭스가 있는 경우 다음을 확인합니다:

1. **R (Responsible)**: 모든 활동에 실행자가 있는가?
2. **A (Accountable)**: 각 활동에 1명만 책임자로 지정되었는가?
3. **C (Consulted)**: 협의 대상이 적절한가?
4. **I (Informed)**: 통보 대상이 누락되지 않았는가?

**검증 규칙**:
- 모든 활동에 최소 R과 A가 있어야 함
- A는 활동당 정확히 1명
- 동일인이 R과 A를 겸할 수 있음

### 4단계: 프로세스 흐름 검증

단계별 절차에서 다음을 확인합니다:

1. **시작 조건**: 프로세스가 시작되는 트리거가 명확한가?
2. **단계 순서**: 논리적으로 연결되는가?
3. **분기 조건**: 조건부 흐름이 명확한가?
4. **완료 조건**: 각 단계와 전체 프로세스의 종료 기준이 있는가?

**체크 항목**:
- [ ] 각 단계에 수행자가 지정됨
- [ ] 각 단계에 트리거/입력이 정의됨
- [ ] 각 단계에 산출물/출력이 정의됨
- [ ] 단계 간 데드엔드(막다른 길)가 없음

### 5단계: 예외 처리 검증

다음을 확인합니다:

1. **예외 상황 식별**: 주요 예외가 정의되었는가?
2. **처리 방법**: 각 예외의 대응 방안이 있는가?
3. **에스컬레이션**: 상위 보고 경로가 명확한가?

### 6단계: 점수 산정

각 항목별 점수를 계산합니다:

| 항목 | 배점 | 기준 |
|------|------|------|
| 구조 완전성 | 3점 | 필수 섹션 존재 여부 |
| RACI 적절성 | 2점 | R/A 지정 및 규칙 준수 |
| 흐름 논리성 | 2점 | 단계 연결 및 조건 명확성 |
| 예외 처리 | 2점 | 예외 식별 및 대응 방안 |
| 측정 가능성 | 1점 | KPI 정의 여부 |
| **총점** | **10점** | |

**점수 해석**:
- 9-10점: 우수 (즉시 사용 가능)
- 7-8점: 양호 (경미한 보완 필요)
- 5-6점: 보통 (주요 보완 필요)
- 0-4점: 미흡 (재작성 권장)

## 출력 형식

분석 결과는 다음 형식으로 출력합니다:

```markdown
## 프로세스 분석 결과: {{docId}}

### 문서 정보

| 항목 | 내용 |
|------|------|
| **docId** | {{docId}} |
| **제목** | {{title}} |
| **Owner** | {{owner}} |
| **상태** | {{status}} |
| **버전** | {{version}} |

---

### 분석 요약

| 항목 | 점수 | 상태 |
|------|------|------|
| 구조 완전성 | X/3 | ✅/⚠️/❌ |
| RACI 적절성 | X/2 | ✅/⚠️/❌ |
| 흐름 논리성 | X/2 | ✅/⚠️/❌ |
| 예외 처리 | X/2 | ✅/⚠️/❌ |
| 측정 가능성 | X/1 | ✅/⚠️/❌ |
| **총점** | **X/10** | **등급** |

---

### 상세 분석

#### 1. 구조 완전성

| 섹션 | 상태 | 설명 |
|------|------|------|
| 목적 | ✅/❌ | {{설명}} |
| 적용 범위 | ✅/❌ | {{설명}} |
| 입력/출력 | ✅/❌ | {{설명}} |
| 역할 (RACI) | ✅/❌ | {{설명}} |
| 프로세스 흐름 | ✅/❌ | {{설명}} |

| KPI | ✅/❌ | {{설명}} |

#### 2. RACI 분석

{{RACI 매트릭스 검증 결과}}

#### 3. 흐름 분석

{{프로세스 흐름 검증 결과}}

#### 4. 예외 처리 분석

---

### 개선 권고

#### 필수 조치 (High)

#### 권장 조치 (Medium)

#### 선택 조치 (Low)

- [ ] 필수 조치 항목 반영
- [ ] governance-auditor 스킬로 거버넌스 검증
- [ ] 상태를 Review로 변경
```

## 예시

### 분석 요청 예시

```
"이 프로세스 문서 분석해줘"
"PRC-planops-001 문제점 찾아줘"
"프로세스 리뷰 부탁해"
```

### 분석 결과 예시

```markdown
## 프로세스 분석 결과: PRC-planops-001

### 문서 정보

| 항목 | 내용 |
|------|------|
| **docId** | PRC-planops-001 |
| **제목** | 연간 사업계획 수립 프로세스 |
| **Owner** | 홍길동 |
| **상태** | Draft |
| **버전** | 0.1 |

---

### 분석 요약

| 항목 | 점수 | 상태 |
|------|------|------|
| 구조 완전성 | 2/3 | ⚠️ |
| RACI 적절성 | 2/2 | ✅ |
| 흐름 논리성 | 1/2 | ⚠️ |
| 예외 처리 | 0/2 | ❌ |
| 측정 가능성 | 1/1 | ✅ |
| **총점** | **6/10** | **보통** |

---

### 개선 권고

#### 필수 조치 (High)
1. 예외 처리 섹션 추가 필요
2. 3단계와 4단계 간 분기 조건 명확화

#### 권장 조치 (Medium)
1. 적용 제외 대상 구체화
```

## 모범 사례

1. **전체 문서 읽기**: 부분이 아닌 전체 문서를 분석
2. **객관적 평가**: 점수 기준을 일관되게 적용
3. **구체적 피드백**: "부족함" 대신 "X가 누락됨"으로 명시
4. **우선순위 제시**: 필수/권장/선택으로 구분
5. **다음 단계 안내**: 분석 후 조치 방향 제시

## 관련 스킬

- **governance-auditor**: 거버넌스 규칙 준수 검증
- **process-designer**: 프로세스 개선/재설계
- **process-documenter**: 분석 결과 반영하여 문서화
