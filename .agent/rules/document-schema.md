# document-schema.md - 문서 체계 규칙

> **상위 규칙**: MASTER.md
> **버전**: 1.0
> **상태**: Draft

---

## 1. 개요

이 규칙은 Process Forge 프로젝트의 **문서 식별자(docId) 네이밍 규칙**과 **문서 유형 체계**를 정의합니다.

---

## 2. docId 네이밍 규칙

### 2.1 기본 형식

```
{TYPE}-{domain}-{seq}
```

| 구성요소 | 설명 | 예시 |
|----------|------|------|
| `TYPE` | 문서 유형 접두사 (대문자) | POL, PRC, RUL, CHG, PROP, RPT |
| `domain` | 업무 도메인 (소문자) | docops, planops, hrops |
| `seq` | 일련번호 (3자리) | 001, 002, 003 |

> **참고**: IDEA 문서는 docId 형식을 따르지 않고 자유 파일명을 사용합니다.

### 2.2 예시

| docId | 설명 |
|-------|------|
| `POL-docops-001` | 문서 운영 정책 #1 |
| `PRC-planops-001` | 계획 운영 프로세스 #1 |
| `RUL-docid-001` | 문서ID 규칙 #1 |
| `CHG-planops-001-001` | PRC-planops-001의 변경요청 #1 |
| `PROP-hrops-001` | 인사 운영 제안서 #1 |
| `RPT-hrops-001` | 인사 운영 검증 리포트 #1 |

### 2.3 CHG (변경요청) 특수 형식

```
CHG-{원본domain}-{원본seq}-{변경seq}
```

예: `CHG-planops-001-001` = PRC-planops-001에 대한 첫 번째 변경요청

## 3. 문서 유형 정의

### 3.1 POL (Policy) - 정책

| 항목 | 내용 |
|------|------|
| **목적** | 조직의 의사결정 기준과 방향 제시 |
| **질문** | "왜(Why) 이렇게 해야 하는가?" |
| **특징** | 추상적, 장기적, 원칙 중심 |
| **저장 위치** | `docs/policies/` |

### 3.2 PRC (Process) - 프로세스

| 항목 | 내용 |
|------|------|
| **목적** | 업무 수행 절차와 흐름 정의 |
| **질문** | "어떻게(How) 수행하는가?" |
| **특징** | 단계별, 순서가 있음, 실행 가능 |
| **저장 위치** | `docs/processes/` |

### 3.3 RUL (Rule) - 규칙

| 항목 | 내용 |
|------|------|
| **목적** | 특정 상황에서 따라야 할 구체적 지침 |
| **질문** | "무엇을(What) 해야 하는가?" |
| **특징** | 구체적, 조건-행동 형태 |
| **저장 위치** | `docs/rules/` |

### 3.4 CHG (Change Request) - 변경요청

| 항목 | 내용 |
|------|------|
| **목적** | Approved 문서의 수정 요청 |
| **질문** | "무엇을 왜 바꾸는가?" |
| **특징** | 변경 사유, 영향도, 승인 필요 |
| **저장 위치** | `docs/changes/` |

### 3.5 IDEA - 아이디어 (비공식)

| 항목 | 내용 |
|------|------|
| **목적** | 자유로운 아이디어 수집 |
| **질문** | "뭘 하면 좋을까?" |
| **특징** | 자유 형식, 파일명 자유, 진입장벽 최소화 |
| **저장 위치** | `docs/ideas/` |
| **파일명** | 자유 (예: `인사개선-아이디어.md`, `새프로세스-구상.md`) |

### 3.6 PROP (Proposal) - 제안서

| 항목 | 내용 |
|------|------|
| **목적** | 여러 아이디어를 통합/구조화한 제안 |
| **질문** | "이렇게 하면 어떨까?" |
| **특징** | 구조화된 형식, Owner 필수 |
| **저장 위치** | `docs/proposals/` |

### 3.7 RPT (Report) - 검증 리포트

| 항목 | 내용 |
|------|------|
| **목적** | 제안서의 표준문서 전환 적합성 검증 결과 |
| **질문** | "전환 준비가 됐는가?" |
| **특징** | 검증 항목, 전환 판정 포함 |
| **저장 위치** | `docs/reports/` |

## 5. 도메인(domain) 명명 규칙

### 5.1 형식
- **소문자** 영문만 사용
- **약어** 권장 (8자 이내)
- **ops** 접미사로 운영 도메인 표시

### 5.2 권장 도메인 예시

| 도메인 | 설명 |
|--------|------|
| `docops` | 문서 운영 |
| `planops` | 계획/진척 운영 |
| `hrops` | 인사 운영 |
| `finops` | 재무 운영 |
| `itops` | IT 운영 |
| `secops` | 보안 운영 |

### 5.3 새 도메인 추가 시
1. 기존 도메인과 중복 여부 확인
2. 약어가 명확한지 검토
3. MASTER.md에 도메인 목록 업데이트

## 6. 문서 간 관계

### 6.1 계층 관계

```
POL (정책)
 └── PRC (프로세스) - 정책을 구현하는 절차
      └── RUL (규칙) - 프로세스 내 세부 지침
```

### 6.2 참조 표기법

문서 내에서 다른 문서 참조 시:

```markdown
관련 문서: [PRC-planops-001](../processes/PRC-planops-001.md)
```

### 6.3 변경 관계

```
원본 문서 ←── CHG (변경요청)
    ↓
수정된 문서 (새 버전)
```

---

## 7. 일련번호(seq) 관리

### 7.1 규칙
- **001**부터 시작
- 동일 TYPE + domain 내에서 순차 증가
- 삭제된 번호는 재사용하지 않음

### 7.2 확인 방법
새 문서 생성 전, 해당 폴더에서 마지막 번호 확인:

```
docs/processes/
├── PRC-planops-001.md
├── PRC-planops-002.md  ← 다음은 003
└── PRC-docops-001.md
```

## 8. 예외 처리

### 8.1 템플릿 파일
- 접두사: `TPL-`
- 위치: `.agent/templates/`
- 예: `TPL-process.md`

### 8.2 메타 규칙 파일
- `.agent/rules/` 내 파일은 docId 없이 직접 명명
- 예: `document-schema.md`, `ownership.md`
