## 1. 개요

### 1.1 목적

본 문서는 **Notion=운영/데이터/알림**, **Obsidian=SSOT/버전/빌드** 원칙 아래에서 팀의 분기(또는 월) 계획을 **한 장(마스터 플랜)**으로 운영하고, **기획(Spec) 일정 + 개발(Dev) 일정**을 일관된 기준으로 진척 관리하기 위한 통합 프로세스를 정의한다.

### 1.2 적용 범위

| 구분 | 내용 |
|------|------|
| 계획 단위 | 분기/월/스프린트 등 "기간(Cycle)" 기반 운영 |
| 대상 | 기획(PM/PO), 개발(Tech Owner), QA, 운영 리드 |
| 제외 | 문서 본문(정책/PRD/상세 프로세스)의 Notion 복제 |

---

## 2. 핵심 원칙

### 2.1 SSOT 분리

| 도구 | 역할 |
|------|------|
| **Notion** | 레코드(상태/담당/기한/리스크/알림/링크) |
| **Obsidian** | 문서 본문(정책/프로세스/PRD/데이터 정의/QA pack) + Git 버전 |

### 2.2 연결키(docId) 강제

- Notion 레코드는 Obsidian 문서와 **docId**로 연결
- 제목 변경은 허용하되 **docId는 불변**
- docId 네이밍은 `.agent/rules/document-schema.md` 규칙 준수

#### docId 네이밍 규칙 (요약)

| 유형 | 형식 | 예시 |
|------|------|------|
| PRD | `PRD-{domain}-{seq}` | PRD-planops-001 |
| QAP | `QAP-{domain}-{seq}` | QAP-planops-001 |
| POL | `POL-{domain}-{seq}` | POL-docops-001 |
| PRC | `PRC-{domain}-{seq}` | PRC-planops-001 |

> 상세: `.agent/rules/document-schema.md` 참조

### 2.3 Single Owner 원칙

- 모든 Phase/Task는 반드시 **1명의 Owner**를 지정
- 공동 Owner 금지 (Deputy 지정은 가능)

### 2.4 변경통제

- Approved 문서는 직접 수정 금지
- 변경은 CHG 문서로 요청 → 승인 → rev 상승
- 상세: `.agent/rules/change-control.md` 참조

## 3. 역할과 책임 (RACI)

| 역할 | 책임 |
|------|------|
| **Cycle Owner(리드)** | Active Cycle 설정, 리플랜 승인, 리스크 최종 조정 |
| **PM/PO(기획 Owner)** | Initiative 정의, Spec Gate 충족, 산출물(docId) 책임 |
| **Tech Owner(개발 Owner)** | Dev 일정/리소스 확인, 기술 리스크/의존성 관리 |
| **QA Owner** | QA/Release Gate 조건 정의 및 검증 |
| **Buddy/Reviewer** | 주간 리뷰, Gate 충족 여부 확인, 품질 피드백 |

## 4. 운영 데이터 모델 (Notion DB 구성)

### 4.1 DB 관계 구조 (3단계 핵심 흐름)

```
[핵심 운영 흐름 - 3단계]
Cycles (1) ── (N) Phases (1) ── (N) Ops Tasks

[포트폴리오 참조 - 선택적]
Phases ──(optional)── Initiatives
```

> **설계 원칙**:
> - 핵심 운영은 Cycle → Phase → Task 3단계로 단순화
> - Initiative는 큰 계획 묶음이 필요할 때만 선택적으로 사용
> - 모든 표준 뷰는 **Cycle Status = Active** 필터 사용 (2단계 Rollup)

### 4.2 구조 단순화 장점

| 항목 | 기존 (4단계) | 개선 (3단계) |
|------|-------------|-------------|
| Rollup 경로 | Task→Phase→Initiative→Cycle | Task→Phase→Cycle |
| Active Cycle 필터 | 3단계 경유 | 2단계 직접 |
| 작은 작업 | Initiative 필수 생성 | Phase만으로 운영 가능 |
| Notion 성능 | Rollup 복잡 | Rollup 단순화 |

### 4.3 DB 요약

| DB | 역할 | 핵심 관계 |
|----|------|----------|
| **Cycles** | 기간 컨테이너 | - |
| **Phases** | 일정의 실체 (마스터 플랜) | Cycle (필수), Initiative (선택) |
| **Ops Tasks** | 주간 실행 티켓 | Phase (필수) |
| **Initiatives** | 큰 계획 묶음 (포트폴리오) | Cycle (필수), Phases에서 역참조 |

## 5. Cycles DB (기간 컨테이너)

### 5.1 필수 속성

| 속성명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| Name | Title | O | 기간 이름 (예: 2026 Q1) |
| Start | Date | O | 기간 시작 |
| End | Date | O | 기간 종료 |
| Status | Select | O | Planned / **Active** / Done |
| Owner | Person | O | 기간 운영 책임자 (단일) |

### 5.2 권장 속성

| 속성명 | 타입 | 설명 |
|--------|------|------|
| Cadence | Select | Quarter / Month / Sprint |
| Goal | Text | 기간의 최종 성과 (1~2문장) |

### 5.3 운영 규칙

- **Active Cycle은 항상 1개만** 유지
- Active가 2개 이상이면 즉시 정리

### 5.4 표준 뷰

| 뷰 이름 | 필터 | 목적 |
|---------|------|------|
| Active Cycle | Status = Active | 현재 운영 기준 |
| Planning Queue | Status = Planned | 다음 기간 준비 |
| Archive | Status = Done | 종료된 기간 |

## 6. Initiatives DB (포트폴리오 - 선택적 사용)

> **역할 변경**: 필수 계층이 아닌 **큰 계획 묶음(포트폴리오)** 역할.
> 작은 작업은 Initiative 없이 Phase만으로 운영 가능.

### 6.1 언제 Initiative를 사용하는가?

| 상황 | Initiative 사용 |
|------|-----------------|
| 여러 Phase를 묶어 관리해야 할 때 | O |
| 분기/월 단위 큰 목표가 있을 때 | O |
| 단순 작업, 버그 수정, 작은 개선 | X (Phase만으로 충분) |
| 독립적인 단일 Phase | X |

### 6.2 필수 속성

| 한글명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| 이니셔티브명 | Title | O | 결과가 드러나게 작성 |
| 사이클 | Relation → Cycles | O | 소속 기간 |
| 상태 | Select | O | 제안됨/확정/진행중/릴리즈됨/제외 |
| 우선순위 | Select | O | P0/P1/P2/P3 |
| 목표(Outcome) | Text | O | 측정 가능한 결과 |
| 기획 책임자 | Person | O | 범위/정책/일정 조율 (단일) |
| 기술 책임자 | Person | O | 기술 판단/리소스 조율 (단일) |
| 메인 docId | Text | O | SSOT 진입점 (PRD) |
| 사이클 상태(롤업) | Rollup | O | 표준 뷰 필터용 |

### 6.3 상태 정의

| 상태 | 설명 |
|------|------|
| 제안됨 (Proposed) | 후보 |
| 확정 (Committed) | 이번 사이클 필수 수행 |
| 진행중 (In Progress) | Spec/Dev 실제 진행 |
| 릴리즈됨 (Released) | 실제 배포/반영 완료 |
| 제외 (Dropped) | 이번 사이클 수행 안 함 |

### 6.4 네이밍 규칙

- 결과가 드러나게 작성
  - O: "정산 수동조정 120→40 감소"
  - X: "정산 고도화"

### 6.5 표준 뷰

| 뷰 이름 | 필터 | 목적 |
|---------|------|------|
| 포트폴리오 | 사이클 상태 = 운영중 | 전체 Initiative 목록 |
| 상태 보드 | 사이클 상태 = 운영중 | 의사결정 흐름 |
| 리스크 레지스터 | 리스크 is not empty | 병목 모아보기 |

## 7. Phases DB (마스터 플랜의 본체)

> **핵심 변경**: Phase가 Cycle에 직접 연결됨 (Initiative 경유 X)

### 7.1 필수 속성

| 한글명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| Phase명 | Title | O | [유형] 작업명 - 목적 |
| **사이클** | **Relation → Cycles** | **O** | **소속 기간 (직접 연결)** |
| 이니셔티브 | Relation → Initiatives | - | 소속 계획 **(선택적)** |
| Phase 유형 | Select | O | Spec/Dev/QA/Release |
| 시작일 | Date | O | 일정 시작 |
| 종료일 | Date | O | 일정 종료 |
| 상태 | Select | O | 미시작/진행중/막힘/완료 |
| 책임자 | Person | O | Phase Owner (단일) |
| Gate 상태 | Select | O | Ready / Not Ready |
| 산출물 docIds | Text | O | SSOT 연결 키 목록 |
| 사이클 상태(롤업) | Rollup | O | **Cycle→Status 직접 (2단계)** |

### 7.1.1 Rollup 경로 변경

| 항목 | 기존 | 개선 |
|------|------|------|
| 사이클 상태 Rollup | Phase→Initiative→Cycle→Status | **Phase→Cycle→Status** |
| Rollup 단계 | 3단계 | **2단계** |

### 7.2 운영 안정화 속성 (권장)

| 속성명 | 타입 | 목적 |
|--------|------|------|
| 막힘 사유 유형 | Select | Blocked 해결 가능하게 |
| 막힘 해소 책임자 | Person | 누가 풀지 명확화 |
| 막힘 해소 목표일 | Date | 언제까지 풀지 |
| 의존 Phase | Relation → Phases | 의존성 추적 |
| 리플랜 변경 여부 | Checkbox | 변경 이력 표시 |

### 7.3 표준 뷰

| 뷰 이름 | 필터 | 목적 |
|---------|------|------|
| **마스터 플랜** | 사이클 상태 = 운영중, Group by Phase 유형 | 한 장에서 전체 확인 |
| Spec 파이프라인 | Phase 유형 = Spec | 기획 관점 |
| Dev 타임라인 | Phase 유형 = Dev | 개발 관점 |
| 막힘 레지스터 | 상태 = 막힘 | 병목 해소 |

|------|------|

## 9. Gate 운영 (품질 관문)

### 9.1 Spec Gate (Ready for Dev) 체크리스트

Dev 착수 전에 Spec Phase가 **Ready**여야 한다.

- [ ] PRD docId 존재 (SSOT 링크 연결됨)
- [ ] 상태/권한/예외 시나리오 포함
- [ ] 데이터 영향(필드/검증 규칙) 명시
- [ ] QA 핵심 TC 또는 QAP docId 링크 존재
- [ ] Tech Owner 리뷰 완료

**운영 룰**
- Gate 미충족 시 Dev Phase를 진행중으로 올리지 않는다
- Dev 중 스펙 변경은 CHG로만 반영

### 9.2 Release Gate 체크리스트

- [ ] QA 완료 (핵심 TC 통과, Sev1/2 없음)
- [ ] 모니터링/로그 포인트 존재
- [ ] 롤백 플랜 존재
- [ ] 릴리즈 노트(docId) 링크

### 9.3 Phase Type별 DoD (완료 정의)

| Phase | 완료 조건 |
|-------|----------|
| Spec | Gate Ready 통과 |
| Dev | main merge + feature flag + 로그/모니터링 포인트 |
| QA | 핵심 TC 통과 + Sev1/2 없음 |
| Release | 배포 완료 + 모니터링 확인 + 롤백 방법 확인 |

## 10. 운영 프로세스 (End-to-End)

### 10.1 Cycle Kickoff (기간 스위치)

**입력:** 새 기간 계획

**절차:**
1. Cycles DB에서 이번 기간 `Status = Active` 설정
2. 기존 Active는 `Done`으로 전환
3. Active Cycle 1개만 존재 확인

**체크리스트:**
- [ ] Active Cycle 1개만 존재
- [ ] Initiatives(확정/옵션) 구분 완료
- [ ] 각 Initiative에 Spec/Dev Phase 생성
- [ ] Phase Start/End 입력 완료
- [ ] Master Plan Timeline 노출 확인

### 10.2 주간 운영 루프

| 요일 | 시간 | 활동 |
|------|------|------|
| 월 | 15분 | 이번 주 Ops Tasks 선정 (최대 5~7개) |
| 수 | 15분 | Blocked/Review Needed 해소 |
| 금 | 30분 | 주간 리뷰 (고정) |

**금요일 리뷰 아젠다:**
1. Master Plan: Blocked Phase 확인
2. Spec Pipeline: Gate 미충족 원인 확인
3. Dev Timeline: 일정 충돌 확인
4. Review Needed: 리뷰 대기 처리
5. 다음 주 Top3 확정 + 리스크 업데이트

### 10.3 Re-plan (월 1회, 첫째 주 금요일)

> 주간 회의에서 범위를 계속 흔들면 계획 신뢰도가 붕괴한다.

**수행 내용:**
- Dropped/Defer 결정
- Phase 일정 재배치 (Start/End 수정)
- 확정 ↔ 옵션 이동
- 의사결정 ADR 기록

**체크리스트:**
- [ ] Dropped/Defer 명확화
- [ ] 일정 재배치 완료
- [ ] ADR 기록 (권장)
- [ ] 이해관계자 커뮤니케이션

### 10.4 Cycle Close (종료 처리)

**절차:**
1. Initiative 상태 정리: 릴리즈됨/이월/제외
2. Carry-over 항목은 다음 Cycle 후보로 이관
3. Cycle Status를 Done으로 전환
4. 회고 요약 (잘된 점/막힌 점/다음 개선)

**체크리스트:**
- [ ] Released/Carry-over 분리
- [ ] 다음 Cycle 후보 등록
- [ ] 회고 요약 작성
- [ ] Cycle Done 전환
