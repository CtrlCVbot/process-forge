# 프로세스 설계자

이 스킬은 새로운 프로세스를 체계적으로 설계하도록 안내합니다.

## 사용 시점

이 스킬을 사용하는 경우:
- 새로운 프로세스 설계
- 기존 프로세스 재설계/개선
- 비정형 업무를 프로세스로 정립
- 워크플로우 설계

## 설계 원칙

프로세스 설계 시 다음 원칙을 따릅니다:

| 원칙 | 설명 |
|------|------|
| **Adoption First** | 따르기 쉬운 프로세스가 좋은 프로세스 |
| **최소 단계** | 불필요한 단계 제거, 핵심에 집중 |
| **명확한 책임** | 모든 활동에 담당자 명시 |
| **측정 가능** | 완료 기준과 KPI 정의 |
| **예외 대비** | 예외 상황과 에스컬레이션 경로 준비 |

## 지침

### 1단계: 요구사항 수집

프로세스 설계 전 다음 정보를 수집합니다:

#### 1.1 필수 질문

| 질문 | 목적 |
|------|------|
| 이 프로세스가 해결할 문제는? | 목적 정의 |
| 누가 이 프로세스를 사용하는가? | 적용 대상 |
| 어떤 상황에서 시작되는가? | 트리거 조건 |
| 최종 결과물은 무엇인가? | 산출물 정의 |
| 관련된 역할/조직은? | 이해관계자 |

#### 1.2 수집 템플릿

```markdown
## 프로세스 요구사항

### 기본 정보
- **프로세스명**:
- **도메인**:
- **요청자**:

### 목적
Q: 이 프로세스가 해결할 문제는?
A:

### 적용 범위
Q: 누가, 언제 이 프로세스를 사용하는가?
A:

### 입력
Q: 프로세스 시작에 필요한 것은?
A:

### 출력
Q: 최종 결과물은?
A:

### 제약 조건
Q: 고려해야 할 제약이 있는가?
A:
```

### 2단계: 프로세스 구조 설계

#### 2.1 입력/출력 정의

| 구분 | 항목 | 설명 | 제공자/수신자 |
|------|------|------|--------------|
| 입력 | {{입력1}} | {{설명}} | {{제공 역할}} |
| 입력 | {{입력2}} | {{설명}} | {{제공 역할}} |
| 출력 | {{출력1}} | {{설명}} | {{수신 역할}} |
| 출력 | {{출력2}} | {{설명}} | {{수신 역할}} |

#### 2.2 핵심 단계 도출

**단계 도출 방법**:

1. **역방향 설계**: 최종 결과물에서 시작하여 역으로 필요한 단계 도출
2. **최소 단계 원칙**: 꼭 필요한 단계만 포함 (3-7단계 권장)
3. **병렬 가능성**: 동시 진행 가능한 단계 식별

**단계 설계 템플릿**:

```markdown
### 단계 X: {{단계명}}

| 항목 | 내용 |
|------|------|
| **수행자** | {{역할}} |
| **트리거** | {{이전 단계 완료 또는 조건}} |
| **활동** | {{수행 내용}} |
| **산출물** | {{결과물}} |
| **완료 조건** | {{다음 단계로 진행 기준}} |
```

### 3단계: RACI 매트릭스 설계

#### 3.1 역할 식별

프로세스에 관련된 모든 역할을 나열합니다:

| 역할 | 설명 |
|------|------|
| {{역할1}} | {{설명}} |
| {{역할2}} | {{설명}} |
| {{역할3}} | {{설명}} |

#### 3.2 RACI 작성 규칙

| 구분 | 의미 | 규칙 |
|------|------|------|
| **R** (Responsible) | 실제 작업 수행 | 복수 가능 |
| **A** (Accountable) | 최종 책임 | **활동당 1명만** |
| **C** (Consulted) | 사전 협의 | 의견 필요 시 |
| **I** (Informed) | 사후 통보 | 결과 공유 필요 시 |

#### 3.3 RACI 템플릿

| 활동 | {{역할1}} | {{역할2}} | {{역할3}} | {{역할4}} |
|------|----------|----------|----------|----------|
| {{활동1}} | R | A | C | I |
| {{활동2}} | I | A | R | - |
| {{활동3}} | - | A | R | I |

**검증 체크리스트**:
- [ ] 모든 활동에 R이 있는가?
- [ ] 모든 활동에 A가 정확히 1명인가?
- [ ] A 없이 R만 있는 활동은 없는가?

### 4단계: 프로세스 흐름 설계

#### 4.1 순차 흐름

```
[시작] → [단계1] → [단계2] → [단계3] → [종료]
```

#### 4.2 분기 흐름

```
[시작] → [단계1] → <조건 판단>
                       ├─ Yes → [단계2A] → [종료A]
                       └─ No  → [단계2B] → [종료B]
```

#### 4.3 병렬 흐름

```
[시작] → [단계1] ─┬→ [단계2A] ─┬→ [단계3] → [종료]
                  └→ [단계2B] ─┘
```

#### 4.4 흐름 검증

| 검증 항목 | 확인 |
|----------|------|
| 시작점이 명확한가? | ✅/❌ |
| 종료점이 명확한가? | ✅/❌ |
| 데드엔드가 없는가? | ✅/❌ |
| 무한 루프가 없는가? | ✅/❌ |
| 분기 조건이 명확한가? | ✅/❌ |

### 7단계: 검증 및 피드백

설계된 프로세스를 검증합니다:

#### 7.1 자체 검증

**Adoption First 체크리스트**:
- [ ] 단계가 7개 이하인가?
- [ ] 각 단계가 명확하고 이해하기 쉬운가?
- [ ] 역할별 책임이 명확한가?
- [ ] 실행자가 추가 설명 없이 따라할 수 있는가?

**완전성 체크리스트**:
- [ ] 목적이 명확한가?
- [ ] 적용 범위가 정의되었는가?
- [ ] 입력/출력이 모두 정의되었는가?
- [ ] RACI가 완성되었는가?
- [ ] 예외 처리가 정의되었는가?
- [ ] KPI가 정의되었는가?

#### 7.2 거버넌스 검증

governance-auditor 스킬을 호출하여 검증:
- docId 형식
- Owner 지정
- 메타데이터 완전성

## 예시

### 설계 요청 예시

```
"신규 입사자 온보딩 프로세스 설계해줘"
"고객 문의 처리 워크플로우 만들어줘"
"문서 승인 프로세스 구성해줘"
```

### 대화형 설계 예시

```
사용자: 문서 승인 프로세스 설계해줘

Claude: 문서 승인 프로세스를 설계하겠습니다.
몇 가지 질문을 드리겠습니다.

1. 어떤 종류의 문서를 승인하는 프로세스인가요?
   (예: 계약서, 기획서, 보고서 등)

2. 승인 단계가 몇 개인가요?
   (예: 1단계, 2단계, 3단계)

3. 승인권자는 누구인가요?
   (예: 팀장, 부서장, 임원 등)

4. 긴급 승인 절차가 필요한가요?

사용자: 기획서 승인이야. 2단계로 팀장 → 부서장.
긴급 승인은 필요해.

Claude: 이해했습니다. 다음과 같이 설계하겠습니다...
```

## 모범 사례

1. **사용자 관점**: 실행자가 따르기 쉬운지 항상 확인
2. **점진적 상세화**: 큰 그림 → 세부 사항 순서
3. **피드백 반영**: 설계 후 이해관계자 검토
4. **표준 준수**: TPL-process.md 템플릿 활용
5. **검증 필수**: governance-auditor로 최종 검증

## 관련 스킬

- **process-analyzer**: 기존 프로세스 분석 후 재설계
- **governance-auditor**: 설계된 프로세스 검증
- **process-documenter**: 설계 결과 문서화
