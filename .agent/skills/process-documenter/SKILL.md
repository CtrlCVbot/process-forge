# 프로세스 문서화 도구

이 스킬은 프로세스 정보를 표준 문서 형식(PRC)으로 작성합니다.

## 사용 시점

이 스킬을 사용하는 경우:
- 구두/메모로 정리된 프로세스를 문서화
- process-designer로 설계한 프로세스를 문서로 작성
- 기존 비정형 문서를 표준 형식으로 변환
- 프로세스 문서 초안 작성

## 지침

### 1단계: 정보 수집

문서화에 필요한 정보를 수집합니다.

#### 1.1 필수 정보 확인

| 항목 | 필수 | 확인 |
|------|------|------|
| 프로세스명 | O | |
| 도메인 (domain) | O | |
| 목적 | O | |
| Owner | O | |
| 적용 대상 | O | |
| 주요 단계 | O | |
| 역할/담당자 | O | |

#### 1.2 정보 수집 질문

정보가 부족한 경우 다음을 질문합니다:

**기본 정보**:
- 프로세스의 이름은 무엇인가요?
- 이 프로세스가 속한 도메인/영역은? (예: planops, hr, finance)
- 문서의 Owner(책임자)는 누구인가요?

**목적/범위**:
- 이 프로세스의 목적은 무엇인가요?
- 누가, 언제 이 프로세스를 사용하나요?
- 적용 제외 대상이 있나요?

**프로세스 내용**:
- 주요 단계는 무엇인가요?
- 각 단계의 담당자는 누구인가요?
- 시작 조건과 종료 조건은?

### 2단계: docId 생성

#### 2.1 docId 형식

```
PRC-{domain}-{seq}
```

| 요소 | 규칙 | 예시 |
|------|------|------|
| TYPE | PRC (프로세스 고정) | PRC |
| domain | 소문자, 하이픈 허용 | planops, hr-recruit |
| seq | 3자리 숫자 | 001, 012 |

#### 2.2 신규 seq 부여

1. 해당 도메인의 기존 문서 확인
2. 가장 큰 seq + 1 부여
3. 충돌 방지를 위해 최신 상태 확인

**예시**:
```
기존: PRC-planops-001, PRC-planops-002
신규: PRC-planops-003
```

### 3단계: 메타데이터 작성

#### 3.1 YAML Frontmatter

```yaml
---
docId: PRC-{{domain}}-{{seq}}
title: {{프로세스 제목}}
owner: {{Owner 이름}}
deputy: {{Deputy 이름}}
status: Draft
version: 0.1
created: {{YYYY-MM-DD}}
updated: {{YYYY-MM-DD}}
---
```

#### 3.2 필드 설명

| 필드 | 필수 | 설명 |
|------|------|------|
| docId | O | 문서 고유 식별자 |
| title | O | 문서 제목 |
| owner | O | 문서 책임자 (개인명) |
| deputy | △ | 대리인 (권장) |
| status | O | Draft로 시작 |
| version | O | 0.1로 시작 |
| created | O | 생성일 (오늘 날짜) |
| updated | O | 수정일 (오늘 날짜) |

### 4단계: 섹션별 작성

TPL-process.md 템플릿을 기반으로 각 섹션을 작성합니다.

#### 4.1 목적 섹션

```markdown
## 1. 목적

이 프로세스의 목적은 {{구체적인 목적}}입니다.
```

**작성 가이드**:
- 한 문장으로 명확하게
- "~하기 위함" 또는 "~하는 것"으로 끝맺음
- 추상적 표현 지양

**좋은 예**: "신규 입사자가 입사 후 2주 내에 업무에 투입될 수 있도록 체계적인 온보딩을 제공하는 것입니다."

**나쁜 예**: "온보딩을 잘하기 위함입니다."

#### 4.2 적용 범위 섹션

```markdown
## 2. 적용 범위

### 2.1 적용 대상

| 구분 | 대상 |
|------|------|
| 조직 | {{적용 조직}} |
| 역할 | {{적용 역할}} |
| 상황 | {{적용 상황}} |

### 2.2 적용 제외

- {{제외 대상/상황}}
```

#### 4.3 입력/출력 섹션

```markdown
## 3. 입력과 출력

### 3.1 입력 (Input)

| 입력 항목 | 설명 | 제공자 |
|----------|------|--------|
| {{입력}} | {{설명}} | {{역할}} |

### 3.2 출력 (Output)

| 출력 항목 | 설명 | 수신자 |
|----------|------|--------|
| {{출력}} | {{설명}} | {{역할}} |
```

#### 4.4 역할 (RACI) 섹션

```markdown
## 4. 역할 (RACI)

| 활동 | R (실행) | A (책임) | C (협의) | I (통보) |
|------|---------|---------|---------| ---------|
| {{활동}} | {{역할}} | {{역할}} | {{역할}} | {{역할}} |
```

**작성 규칙**:
- R: 실제 작업 수행자
- A: 최종 책임자 (활동당 1명만)
- C: 사전 협의 필요 대상
- I: 사후 통보 대상

#### 4.5 프로세스 흐름 섹션

```markdown
## 5. 프로세스 흐름

### 5.1 개요 다이어그램

[시작] → [단계1] → [단계2] → [단계3] → [종료]

### 5.2 상세 단계

#### 단계 1: {{단계명}}

| 항목 | 내용 |
|------|------|
| **수행자** | {{역할}} |
| **트리거** | {{시작 조건}} |
| **활동** | {{수행 내용}} |
| **산출물** | {{결과물}} |
| **완료 조건** | {{종료 기준}} |

**상세 절차**:
1. {{세부 절차 1}}
2. {{세부 절차 2}}
3. {{세부 절차 3}}
```

#### 4.6 예외 처리 섹션

```markdown
## 6. 예외 처리

### 6.1 예외 상황

| 예외 상황 | 발생 조건 | 처리 방법 |
|----------|----------|----------|
| {{예외}} | {{조건}} | {{방법}} |

### 6.2 에스컬레이션

| 수준 | 조건 | 대상 | 대응 시간 |
|------|------|------|----------|
| Level 1 | {{조건}} | {{담당자}} | {{시간}} |
| Level 2 | {{조건}} | {{관리자}} | {{시간}} |
```

#### 4.7 KPI 섹션

```markdown
## 7. 측정 지표 (KPI)

| 지표 | 정의 | 목표 | 측정 주기 |
|------|------|------|----------|
| {{지표명}} | {{정의}} | {{목표값}} | {{주기}} |
```

#### 4.8 관련 문서 섹션

```markdown
## 8. 관련 문서

### 8.1 상위 문서

| 유형 | docId | 제목 |
|------|-------|------|
| 정책 | {{POL-xxx-xxx}} | {{제목}} |

### 8.2 하위 문서

| 유형 | docId | 제목 |
|------|-------|------|
| 규칙 | {{RUL-xxx-xxx}} | {{제목}} |

### 8.3 참조 문서

- {{참조 문서}}
```

#### 4.9 변경 이력 섹션

```markdown
## 9. 변경 이력

| 버전 | 날짜 | 변경 내용 | CHG | 작성자 |
|------|------|----------|-----|--------|
| 0.1 | {{YYYY-MM-DD}} | 최초 작성 | - | {{owner}} |
```

### 5단계: 검증

작성된 문서를 검증합니다.

#### 5.1 완전성 체크

| 섹션 | 필수 | 작성 |
|------|------|------|
| 메타데이터 | O | ✅/❌ |
| 목적 | O | ✅/❌ |
| 적용 범위 | O | ✅/❌ |
| 입력/출력 | O | ✅/❌ |
| RACI | O | ✅/❌ |
| 프로세스 흐름 | O | ✅/❌ |
| 예외 처리 | O | ✅/❌ |
| KPI | △ | ✅/❌ |
| 관련 문서 | △ | ✅/❌ |
| 변경 이력 | O | ✅/❌ |

#### 5.2 거버넌스 검증

governance-auditor 스킬을 호출하여 검증:
- docId 형식 준수
- Owner 규칙 준수
- 메타데이터 완전성

### 6단계: 저장 및 안내

#### 6.1 파일명 규칙

```
파일명: {docId}.md
예: PRC-planops-001.md
```

#### 6.2 저장 위치

```
프로젝트루트/docs/processes/PRC-planops-001.md
```

#### 6.3 다음 단계 안내

```markdown
## 문서화 완료

**생성된 문서**: {{docId}}
**파일**: {{파일 경로}}

### 다음 단계

1. **검토 요청**: 문서 내용을 이해관계자에게 검토 요청
2. **상태 변경**: 검토 완료 후 Draft → Review
3. **승인 요청**: Review 완료 후 Review → Approved

### 참고

- 문서 수정: Draft 상태에서 자유롭게 수정 가능
- Approved 후 수정: CHG 문서 작성 필요
```

## 출력 형식

완성된 문서는 TPL-process.md 템플릿 형식으로 출력합니다.

## 예시

### 문서화 요청 예시

```
"이 프로세스 문서화해줘"
"메모 내용을 PRC 문서로 정리해줘"
"프로세스 문서 초안 만들어줘"
```

### 대화형 문서화 예시

```
사용자: 팀 회의 프로세스를 문서화해줘

Claude: 팀 회의 프로세스를 문서화하겠습니다.
몇 가지 정보가 필요합니다.

1. 이 프로세스가 속한 도메인은 무엇인가요?
   (예: teamwork, communication 등)

2. 문서의 Owner(책임자)는 누구인가요?

3. 회의의 주요 단계를 알려주세요.
   (예: 안건 수집 → 회의 진행 → 회의록 작성)

사용자: 도메인은 teamwork, Owner는 김팀장.
단계는 안건수집 → 회의소집 → 회의진행 → 회의록작성 → 공유

Claude: 다음과 같이 문서를 작성하겠습니다.

docId: PRC-teamwork-001
title: 팀 회의 프로세스
...
```

## 모범 사례

1. **정보 먼저 수집**: 문서화 전 충분한 정보 확보
2. **템플릿 준수**: TPL-process.md 형식 유지
3. **플레이스홀더 제거**: {{}} 형태 모두 실제 값으로 대체
4. **일관된 용어**: 동일 개념에 동일 용어 사용
5. **검증 필수**: 저장 전 governance-auditor로 검증

## 관련 스킬

- **process-designer**: 프로세스 설계 후 문서화
- **process-analyzer**: 기존 문서 분석 후 개선
- **governance-auditor**: 문서 검증
