title: 운영/진척 관리 프로세스 (Active Cycle 기반 Quarter/Montly Master Plan)


# 1. 목적 (Purpose)

본 문서는 **Notion=운영/데이터/알림**, **Obsidian=SSOT/버전/빌드** 원칙 아래에서

팀의 분기(또는 월) 계획을 **한 장(마스터 플랜)**으로 운영하고, 동시에 **기획(Spec) 일정 + 개발(Dev) 일정**을 일관된 기준으로 진척 관리하기 위한 표준 프로세스를 정의한다.

# 2. 범위 (Scope)

- 계획 단위: 분기/월/스프린트 등 “기간(Cycle)” 기반 운영 전반
- 대상: 기획(PM/PO), 개발(Tech Owner), QA, 운영 리드
- 제외: 문서 본문(정책/PRD/상세 프로세스)의 Notion 복제(금지)

# 3. 핵심 원칙 (Separation Principles)

## 3.1 SSOT 분리

- **Notion**: 레코드(상태/담당/기한/리스크/알림/링크)
- **Obsidian**: 문서 본문(정책/프로세스/PRD/데이터 정의/QA pack) + Git 버전 + 빌드 배포

## 3.2 연결키(docId) 강제

- Notion의 각 레코드는 Obsidian 문서와 **docId**로 연결한다.
- 제목 변경은 허용하되 **docId는 불변**을 원칙으로 한다.

## 3.3 Approved 변경통제

- Approved 문서는 직접 수정하지 않는다.
- 변경은 CHG 문서로 요청 → 승인 → rev 상승으로 반영한다. (자세한 사항은 `POL-docops-001` 준수)

# 4. 역할과 책임 (Roles & RACI)

| 역할 | 책임 |
| --- | --- |
| Cycle Owner(리드) | Active Cycle 설정, 리플랜 승인, 리스크 최종 조정 |
| PM/PO(기획 Owner) | Initiative 정의, Spec Gate 충족, 산출물(docId) 책임 |
| Tech Owner(개발 Owner) | Dev 일정/리소스 확인, 기술 리스크/의존성 관리 |
| QA Owner | QA/Release Gate 조건 정의 및 검증 |
| Buddy/Reviewer | 주간 리뷰, Gate 충족 여부 확인, 품질 피드백 |

# 5. 운영 데이터 모델 (Notion DB 구성)

> 모든 마스터 플랜/진척 화면은 **Cycle Status = Active**만 바라본다.
> 
> 
> 분기→월로 바뀌어도 필터/뷰는 깨지지 않는다.
> 

## 5.1 DB 목록 (권장 4개)

1. **Cycles**: 기간 컨테이너(분기/월/스프린트)
2. **Initiatives**: 큰 계획(에픽/이니셔티브)
3. **Phases**: 일정의 실체(Spec/Dev/QA/Release)
4. **Ops Tasks**: 주간 실행 티켓(사람이 실제 수행)

## 5.2 관계 구조 (요약)

- Cycle 1 — N Initiatives
- Initiative 1 — N Phases
- Phase 1 — N Ops Tasks

## 5.3 필수 속성(요약)

### A) Cycles

- Name, Start, End
- Status: Planned / **Active** / Done
- Owner
- (선택) Cadence: Quarter / Month / Sprint

### B) Initiatives

- Cycle (relation)
- Status: Proposed / Committed / In Progress / Released / Dropped
- Priority
- Owner(PM), Tech Owner
- Main docId (PRD)
- Risks/Dependencies
- **Cycle Status**(rollup: Cycle → Status) ✅

### C) Phases (마스터 플랜의 본체)

- Initiative (relation)
- Phase Type: Spec / Dev / QA / Release
- Start, End
- Owner
- Status: Not Started / In Progress / Blocked / Done
- Gate: Ready / Not Ready
- DocIds(텍스트): 관련 산출물 docId 목록
- **Cycle Status**(rollup: Initiative → Cycle → Status) ✅

### D) Ops Tasks

- Phase (relation)
- Status: Backlog / This Week / In Progress / Review Needed / Done / Parking
- Owner, Due
- Deliverable docIds, SSOT Link
- DoD 체크
- **Cycle Status**(rollup: Phase → Cycle Status) ✅

# 6. 표준 뷰 (모든 뷰 필터 = Cycle Status Active)

## 6.1 Quarter(또는 Monthly) Master Plan — “한 장”

- DB: **Phases**
- View: Timeline
- Filter: `Cycle Status = Active`
- Group: `Phase Type` (Spec/Dev/QA/Release)
- 목적: **기획 일정 + 개발 일정 + QA/릴리즈**를 한 화면에서 확인

## 6.2 Spec Pipeline (기획자 판)

- DB: Phases / Board + Timeline
- Filter: `Cycle Status = Active` AND `Phase Type = Spec`

## 6.3 Dev Timeline (개발 일정 판)

- DB: Phases / Timeline
- Filter: `Cycle Status = Active` AND `Phase Type = Dev`

## 6.4 Weekly Execution (주간 실행 판)

- DB: Ops Tasks / Kanban
- Filter: `Cycle Status = Active` AND (이번 주 due 또는 This Week/In Progress/Review Needed)

## 6.5 Initiative Portfolio (큰 계획 리스트)

- DB: Initiatives / Table
- Filter: `Cycle Status = Active`

# 7. 운영 프로세스 (End-to-End)

## 7.1 Cycle Kickoff (기간 스위치)

**입력:** 새 기간 계획(분기/월)

**행동:**

1. Cycles DB에서 이번 기간의 `Status = Active`로 설정
2. 기존 Active는 `Done`으로 전환
3. 모든 표준 뷰가 자동으로 “이번 기간”으로 전환됨**출력:** Active Cycle 1개 확정

**체크포인트**

- Active Cycle이 2개 이상이면 운영 혼선 발생 → 즉시 정리

---

## 7.2 계획 편성 (Initiatives → Phases 생성)

**목표:** 큰 계획이 “실제 일정(Phase)”로 내려오게 만들기

1. Initiatives 등록/정리
    - Committed(확정) / Stretch(옵션) 구분 권장
2. Initiative마다 Phases 최소 2개 생성
    - Spec Phase(스펙 기간)
    - Dev Phase(개발 기간)
    - 필요 시 QA/Release 추가
3. 각 Phase에 **Start/End**를 반드시 부여(마스터 플랜 가시화)

**출력:** Master Plan Timeline에 스펙/개발 일정이 노출

---

## 7.3 Spec Gate (Ready for Dev) — 일정 신뢰 장치

Dev 착수 전에 Spec Phase가 **Ready**여야 한다.

### 최소 Gate 조건(권장)

- PRD docId 존재(SSOT 링크 연결)
- 상태/권한/예외 고려 포함
- 데이터 영향(필드/검증) 명시
- QA 핵심 TC 링크 존재(또는 QAP docId)

**운영 룰**

- Gate 미충족 상태에서 Dev Phase를 In Progress로 올리지 않는다.
- Dev 진행 중 스펙 변경은 CHG로만(Approved/합의된 내용 보호)

---

## 7.4 주간 운영 루프 (Weekly Cadence)

### 주간 리듬(권장)

- 월(15분): 이번 주 Ops Tasks 선정(최대 5~7개)
- 수(15분): Blocked/Review Needed 해소
- 금(30분): 주간 리뷰(고정)

### 금요일 30분 리뷰 아젠다(표준)

1. **Master Plan(Phases Timeline)**: Blocked Phase 확인
2. **Spec Pipeline**: Gate 미충족 원인/지원 필요 확인
3. **Dev Timeline**: 일정 충돌/과부하 확인
4. **Review Needed(Ops Tasks)**: 리뷰 대기 티켓 처리
5. 다음주 Top3 확정 + 리스크/의존성 업데이트

**출력:** 다음 주 실행 티켓(Ops Tasks) 확정, 리스크 갱신

---

## 7.5 Re-plan (격주 또는 월 1회만)

주간 회의에서 범위를 계속 흔들면 계획 신뢰도가 붕괴한다.

따라서 **리플랜은 정해진 슬롯에서만** 수행한다.

**Re-plan에서 하는 것**

- Dropped/Defer 결정
- Phase 일정 재배치(Start/End 수정)
- Committed ↔ Stretch 이동
- 주요 의사결정은 ADR로 남김(권장)

---

## 7.6 Release Gate (릴리즈 통과조건)

Release Phase(또는 Release 체크포인트)에서 아래를 확인한다.

- QA 완료(핵심 TC 통과)
- 모니터링/로그 포인트 존재
- 롤백 플랜 존재
- 릴리즈 노트(docId) 링크

**출력:** 릴리즈 승인 또는 보류(리스크 명시)

---

## 7.7 Cycle Close (종료 처리)

- Initiative 상태 정리: Released / Not Shipped / Dropped
- Carry-over(이월) 항목은 다음 Cycle 후보로 이관
- Cycle Status를 Done으로 전환
- 회고(핵심 3개): 잘된 점/막힌 점/다음 개선

# 8. 변경통제 (Change Control)

- Approved 문서/정책/PRD 변경은 CHG로만 처리한다.
- CHG 승인 후:
    - Obsidian 문서 rev 상승
    - Notion Docs Registry 상태/Rev/Link 업데이트
    - 관련 Phase/Task에 영향 표시(필요 시 일정 재조정은 Re-plan 슬롯에서)

# 9. 운영 체크리스트 (실무용)

## 9.1 Kickoff 체크리스트

- [ ]  Active Cycle 1개만 존재
- [ ]  Initiatives(Committed/Stretch) 구분 완료
- [ ]  각 Initiative에 Spec/Dev Phase 생성
- [ ]  Phase Start/End 입력 완료
- [ ]  Master Plan Timeline에서 전체 일정 노출 확인

## 9.2 주간 운영 체크리스트(금요일)

- [ ]  Blocked Phase 원인/담당 지정
- [ ]  Gate 미충족 항목의 “누가/언제” 확정
- [ ]  Review Needed 티켓 처리(승인/반려/수정)
- [ ]  다음 주 Top3(최대 3) 확정
- [ ]  리스크/의존성 업데이트

## 9.3 Re-plan 체크리스트

- [ ]  Dropped/Defer 명확화
- [ ]  일정 재배치(Phases Start/End)
- [ ]  의사결정 ADR 기록(권장)
- [ ]  이해관계자 커뮤니케이션 완료

## 9.4 Close 체크리스트

- [ ]  Released/Carry-over 분리
- [ ]  다음 Cycle 후보 Seed 등록
- [ ]  회고 요약(3줄) 작성
- [ ]  Cycle Done 전환

# 10. 지표(권장, 운영 안정화용)

- 계획 대비 완료율(Committed 기준)
- Carry-over 비율(이월률)
- Blocked 시간(Phase/Task)
- Spec Gate 리드타임(초안→Ready)
- Review SLA(Review Needed 평균 대기시간)

# 11. 부록: 표준 산출물 연결 규칙

- Notion 레코드에는 **docId + 링크 + 상태**만
- 문서 본문은 Obsidian에만 존재
- docId가 없는 산출물은 “완료”로 인정하지 않는다(운영 룰)
